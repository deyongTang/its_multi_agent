# V3.3 升级设计方案：Native KNN + Cross-Encoder Reranking

> **目标**：提升检索性能与精度，为 LLM 提供更高质量的上下文。
> **原则**：最小改动范围，不破坏现有接口，向后兼容。

---

## 一、现状分析

### 1.1 好消息：Mapping 已就绪，无需重建索引

检查 `scripts/init_es_index.py`，`content_vector` 字段已正确配置：

```python
"content_vector": {
    "type": "dense_vector",
    "dims": settings.ES_VECTOR_DIMS,  # 1024
    "index": True,                    # ✅ 已开启 HNSW 索引
    "similarity": "cosine"            # ✅ 余弦相似度
}
```

**结论**：KNN 升级只需修改查询 DSL，**无需重建索引，无需迁移数据**。

### 1.2 问题根因

| 问题 | 位置 | 表现 |
|------|------|------|
| 向量检索 O(n) 全量扫描 | `_vector_search()` | 使用 `script_score` + `match_all`，对每条文档计算余弦相似度 |
| 无重排序 | `retrieve()` 末尾 | RRF 融合后直接返回，未对 query-doc 对做精细打分 |

## 二、方案一：向量检索升级为 Native KNN

### 2.1 改动原理

当前 `_vector_search` 的 DSL：
```python
# 现状：script_score 全量扫描 O(n)
{
    "query": {
        "bool": {
            "filter": [{"term": {"doc_type": "chunk"}}],
            "must": [{
                "script_score": {
                    "query": {"match_all": {}},  # 全量扫描
                    "script": {
                        "source": "cosineSimilarity(params.query_vector, 'content_vector') + 1.0"
                    }
                }
            }]
        }
    }
}
```

改为 ES 原生 `knn` 查询（HNSW，O(log n)）：
```python
# 目标：knn 顶级查询 O(log n)
{
    "knn": {
        "field": "content_vector",
        "query_vector": query_vector,
        "k": top_k,
        "num_candidates": top_k * 10,   # 候选集倍数，影响精度与速度的平衡
        "filter": {"term": {"doc_type": "chunk"}}  # 同样只检索 chunk
    },
    "size": top_k,
    "_source": ["doc_id", "knowledge_no", "title", "content", "chunk_index"]
}
```

### 2.2 参数说明

| 参数 | 含义 | 推荐值 |
|------|------|--------|
| `k` | 最终返回的 KNN 结果数 | `top_k * 3`（与 BM25 对齐，供 RRF 使用） |
| `num_candidates` | HNSW 探索的候选节点数，越大越准但越慢 | `k * 10`（平衡点） |
| `filter` | 前置过滤，只在 chunk 文档中搜索 | `{"term": {"doc_type": "chunk"}}` |

### 2.3 改动范围

```
services/es_retrieval_service.py
  └── _vector_search()         ← 唯一需要改动的方法
      将 script_score DSL 替换为 knn DSL

（hybrid_search() 中的 script_score 同步替换，但主路径是 rrf_search()）
```

`rrf_search()` 流程**不变**，只是底层 `_vector_search()` 从暴力计算换成了 HNSW 索引查询：

```
rrf_search()
  ├── _keyword_search()   BM25，不变
  └── _vector_search()    ← 只改这里的 DSL
        ↓
  _rrf_fusion()           不变
        ↓
  Top-K 结果              不变
```

### 2.4 兼容性确认

- ES 版本要求：8.0+（当前已是 8.x）✅
- Mapping 要求：`dense_vector` + `index: true`（已满足）✅
- 现有数据：无需重新入库 ✅
- 接口变化：无，`_vector_search()` 入参和返回值不变 ✅

## 三、方案二：Cross-Encoder 重排序

### 3.1 集成位置

重排序插入在 **RRF 融合之后、获取父文档之前**：

```
现状：
  rrf_search() → Top-K chunks → get_parent_documents() → LLM

升级后：
  rrf_search() → Top-N candidates → Reranker → Top-K chunks → get_parent_documents() → LLM
                    (N=50)                         (K=5)
```

**为什么在获取父文档之前**：父文档是完整内容，体积大。先用 chunk 做重排序，截断到 Top-K 后再批量取父文档，减少不必要的 mget 开销。

### 3.2 模型选择

> ⚠️ **重要：本地模型不适合生产环境**
>
> PyTorch 推理是 CPU 密集型同步操作。在 Docker 容器（无 GPU）下，并发请求会导致：
> CPU 100% → 队列积压 → 响应超时 → 服务崩溃。
> Python GIL 也不允许多线程真正并行跑 CPU 密集计算，`run_in_executor` 无法解决此问题。
> **生产环境必须使用 API，禁止在主服务中直接加载模型。**

| 方案 | 服务商 | 模型 | 适用环境 | 推荐度 |
|------|--------|------|---------|--------|
| **A. 硅基流动 Rerank API** | 硅基流动 | `BAAI/bge-reranker-v2-m3` | ✅ 生产环境首选 | ⭐⭐⭐ |
| **B. 阿里百炼 Rerank API** | 阿里百炼 | `gte-rerank` | 备选 | ⭐⭐ |
| **C. 本地 BGE Reranker** | 本地 | `bge-reranker-base` | 本地开发 / 调试 | ⭐⭐ |
| **D. 独立推理微服务（GPU）** | 自建 | `bge-reranker-large` | 有专用 GPU 服务器时 | ⭐⭐ |
| **E. Cohere Rerank API** | Cohere | `rerank-multilingual-v3.0` | ❌ 需境外网络，不推荐 | ⭐ |

**推荐方案 A（硅基流动 `BAAI/bge-reranker-v2-m3`）**，理由：
1. `bge-reranker-v2-m3` 是 BGE 系列最新多语言重排序模型，中文效果优秀
2. 硅基流动提供 OpenAI 兼容接口，调用方式与现有 LLM/Embedding 完全一致
3. API 服务端处理并发，主服务无推理压力，不影响吞吐量
4. 无需维护模型文件和推理环境，Docker 镜像保持轻量
5. 按调用量付费，价格相比阿里百炼更低

**硅基流动 Rerank API 调用方式：**
```python
# 硅基流动兼容 OpenAI 格式，Base URL 替换即可
POST https://api.siliconflow.cn/v1/rerank
Authorization: Bearer {SILICONFLOW_API_KEY}

{
    "model": "BAAI/bge-reranker-v2-m3",
    "query": "用户问题",
    "documents": ["文档1内容", "文档2内容", ...],
    "top_n": 5,
    "return_documents": false
}
```

### 3.3 新增文件设计

**新建 `services/reranker_service.py`**：

```
RerankerService
  ├── __init__(model_name, device)    加载模型，支持 cpu/cuda
  ├── rerank(query, docs, top_k)      核心方法：输入查询+候选集，输出重排后的 Top-K
  └── _score_pairs(pairs)             批量计算 (query, doc) 相关性分数
```

接口约定（与现有 `es_retrieval_service.py` 对齐）：
```python
def rerank(
    self,
    query: str,
    docs: List[Dict[str, Any]],   # 与 rrf_search 返回结构相同
    top_k: int = 5
) -> List[Dict[str, Any]]:        # 返回结构相同，新增 rerank_score 字段
```

### 3.4 改动范围

```
services/
  └── reranker_service.py          ← 新增（封装百炼 API 调用）

services/es_retrieval_service.py
  └── retrieve()                   ← 注入 RerankerService，在 rrf_fusion 后调用

config/settings.py
  ├── SILICONFLOW_API_KEY          ← 新增：硅基流动 API Key
  ├── RERANKER_BASE_URL            ← 新增：https://api.siliconflow.cn/v1
  ├── RERANKER_MODEL               ← 新增：BAAI/bge-reranker-v2-m3
  ├── RERANKER_TOP_N               ← 新增：送入 reranker 的候选数（默认 50）
  └── RERANKER_ENABLED             ← 新增：开关，方便降级

requirements.txt
  └── 无需新增（复用现有 openai/httpx 客户端调用百炼 API）
```

### 3.5 `retrieve()` 方法升级后的完整流程

```python
def retrieve(query, top_k=5, return_full_content=True):
    # Step 1: 混合检索，多召回候选（top_n >> top_k）
    top_n = settings.RERANKER_TOP_N  # 50
    search_results = rrf_search(query, top_k=top_n)

    # Step 2: Reranker 精排（可开关）
    if settings.RERANKER_ENABLED and search_results:
        search_results = reranker_service.rerank(query, search_results, top_k=top_k)
    else:
        search_results = search_results[:top_k]

    # Step 3: 获取父文档（只取 top_k 条，减少 mget 开销）
    if return_full_content:
        knowledge_nos = [r["knowledge_no"] for r in search_results]
        parent_docs = get_parent_documents(knowledge_nos)
        for result in search_results:
            result["full_content"] = parent_docs.get(result["knowledge_no"], "")

    return search_results
```

### 3.6 Reranker 降级策略

`RERANKER_ENABLED = False` 时，系统自动退化为当前 RRF 行为，不影响任何现有功能。

## 四、升级后完整检索流程

```
用户问题
    ↓
[Query Rewriting]  →  原始 + 改写版本
    ↓
[multi_query_rrf_search]
    ├── BM25 检索 × 2 (原始 + 改写)
    └── KNN 检索 × 2  ← 改为 Native KNN
    ↓
[RRF Fusion]  →  Top 50 候选（扩大候选集）
    ↓
[Cross-Encoder Reranker]  ← 新增
    │   输入：(query, chunk_content) × 50
    │   输出：相关性分数 0~1
    ↓
[Top-K Truncate]  →  Top 5
    ↓
[Get Parent Documents]  →  完整内容
    ↓
[LLM Generation]  →  流式输出
```

## 六、风险与注意事项

| 风险点 | 影响 | 应对措施 |
|--------|------|---------|
| ~~本地模型并发崩溃~~ | ~~CPU 100%，服务挂死~~ | **已规避：生产使用百炼 API，禁止主服务加载本地模型** |
| Reranker API 网络延迟 | +50~200ms | 控制候选集不超过 50 条；`RERANKER_ENABLED` 开关可随时关闭降级 |
| 百炼 API 调用费用 | 增加运营成本 | Rerank 调用量 = 查询量，按需评估；可设置 Top-N 上限控制成本 |
| KNN `num_candidates` 设置过小 | 召回率下降 | 建议不低于 `k * 5`，初期设为 `k * 10` 观察效果 |
| 现有 `hybrid_search()` 未更新 | 该方法仍用 script_score | 主路径已是 `rrf_search()`，`hybrid_search()` 同步替换即可 |
