# V3.3 升级设计方案：Native KNN + Cross-Encoder Reranking

> **目标**：提升检索性能与精度，为 LLM 提供更高质量的上下文。
> **原则**：最小改动范围，不破坏现有接口，向后兼容。

---

## 一、现状分析

### 1.1 好消息：Mapping 已就绪，无需重建索引

检查 `scripts/init_es_index.py`，`content_vector` 字段已正确配置：

```python
"content_vector": {
    "type": "dense_vector",
    "dims": settings.ES_VECTOR_DIMS,  # 1024
    "index": True,                    # ✅ 已开启 HNSW 索引
    "similarity": "cosine"            # ✅ 余弦相似度
}
```

**结论**：KNN 升级只需修改查询 DSL，**无需重建索引，无需迁移数据**。

### 1.2 问题根因

| 问题 | 位置 | 表现 |
|------|------|------|
| 向量检索 O(n) 全量扫描 | `_vector_search()` | 使用 `script_score` + `match_all`，对每条文档计算余弦相似度 |
| 无重排序 | `retrieve()` 末尾 | RRF 融合后直接返回，未对 query-doc 对做精细打分 |

## 二、方案一：向量检索升级为 Native KNN

### 2.1 改动原理

当前 `_vector_search` 的 DSL：
```python
# 现状：script_score 全量扫描 O(n)
{
    "query": {
        "bool": {
            "filter": [{"term": {"doc_type": "chunk"}}],
            "must": [{
                "script_score": {
                    "query": {"match_all": {}},  # 全量扫描
                    "script": {
                        "source": "cosineSimilarity(params.query_vector, 'content_vector') + 1.0"
                    }
                }
            }]
        }
    }
}
```

改为 ES 原生 `knn` 查询（HNSW，O(log n)）：
```python
# 目标：knn 顶级查询 O(log n)
{
    "knn": {
        "field": "content_vector",
        "query_vector": query_vector,
        "k": top_k,
        "num_candidates": top_k * 10,   # 候选集倍数，影响精度与速度的平衡
        "filter": {"term": {"doc_type": "chunk"}}  # 同样只检索 chunk
    },
    "size": top_k,
    "_source": ["doc_id", "knowledge_no", "title", "content", "chunk_index"]
}
```

### 2.2 参数说明

| 参数 | 含义 | 推荐值 |
|------|------|--------|
| `k` | 最终返回的 KNN 结果数 | `top_k * 3`（与 BM25 对齐，供 RRF 使用） |
| `num_candidates` | HNSW 探索的候选节点数，越大越准但越慢 | `k * 10`（平衡点） |
| `filter` | 前置过滤，只在 chunk 文档中搜索 | `{"term": {"doc_type": "chunk"}}` |

### 2.3 改动范围

```
services/es_retrieval_service.py
  └── _vector_search()         ← 唯一需要改动的方法
      将 script_score DSL 替换为 knn DSL

（hybrid_search() 中的 script_score 同步替换，但主路径是 rrf_search()）
```

`rrf_search()` 流程**不变**，只是底层 `_vector_search()` 从暴力计算换成了 HNSW 索引查询：

```
rrf_search()
  ├── _keyword_search()   BM25，不变
  └── _vector_search()    ← 只改这里的 DSL
        ↓
  _rrf_fusion()           不变
        ↓
  Top-K 结果              不变
```

### 2.4 兼容性确认

- ES 版本要求：8.0+（当前已是 8.x）✅
- Mapping 要求：`dense_vector` + `index: true`（已满足）✅
- 现有数据：无需重新入库 ✅
- 接口变化：无，`_vector_search()` 入参和返回值不变 ✅

## 三、方案二：Cross-Encoder 重排序

### 3.1 集成位置

重排序插入在 **RRF 融合之后、获取父文档之前**：

```
现状：
  rrf_search() → Top-K chunks → get_parent_documents() → LLM

升级后：
  rrf_search() → Top-N candidates → Reranker → Top-K chunks → get_parent_documents() → LLM
                    (N=50)                         (K=5)
```

**为什么在获取父文档之前**：父文档是完整内容，体积大。先用 chunk 做重排序，截断到 Top-K 后再批量取父文档，减少不必要的 mget 开销。

### 3.2 模型选择

> ⚠️ **重要：本地模型不适合生产环境**
>
> PyTorch 推理是 CPU 密集型同步操作。在 Docker 容器（无 GPU）下，并发请求会导致：
> CPU 100% → 队列积压 → 响应超时 → 服务崩溃。
> Python GIL 也不允许多线程真正并行跑 CPU 密集计算，`run_in_executor` 无法解决此问题。
> **生产环境必须使用 API，禁止在主服务中直接加载模型。**

| 方案 | 服务商 | 模型 | 适用环境 | 推荐度 |
|------|--------|------|---------|--------|
| **A. 硅基流动 Rerank API** | 硅基流动 | `BAAI/bge-reranker-v2-m3` | ✅ 生产环境首选 | ⭐⭐⭐ |
| **B. 阿里百炼 Rerank API** | 阿里百炼 | `gte-rerank` | 备选 | ⭐⭐ |
| **C. 本地 BGE Reranker** | 本地 | `bge-reranker-base` | 本地开发 / 调试 | ⭐⭐ |
| **D. 独立推理微服务（GPU）** | 自建 | `bge-reranker-large` | 有专用 GPU 服务器时 | ⭐⭐ |
| **E. Cohere Rerank API** | Cohere | `rerank-multilingual-v3.0` | ❌ 需境外网络，不推荐 | ⭐ |

**推荐方案 A（硅基流动 `BAAI/bge-reranker-v2-m3`）**，理由：
1. `bge-reranker-v2-m3` 是 BGE 系列最新多语言重排序模型，中文效果优秀
2. 硅基流动提供 OpenAI 兼容接口，调用方式与现有 LLM/Embedding 完全一致
3. API 服务端处理并发，主服务无推理压力，不影响吞吐量
4. 无需维护模型文件和推理环境，Docker 镜像保持轻量
5. 按调用量付费，价格相比阿里百炼更低

**硅基流动 Rerank API 调用方式：**
```python

POST https://api.siliconflow.cn/v1/rerank
Authorization: Bearer {SILICONFLOW_API_KEY}
```

### 3.5 `retrieve()` 方法升级后的完整流程

```python
def retrieve(query, top_k=5, return_full_content=True):
    # Step 1: 混合检索，多召回候选（top_n >> top_k）
    top_n = settings.RERANKER_TOP_N  # 50
    search_results = rrf_search(query, top_k=top_n)

    # Step 2: Reranker 精排（可开关）
    if settings.RERANKER_ENABLED and search_results:
        search_results = reranker_service.rerank(query, search_results, top_k=top_k)
    else:
        search_results = search_results[:top_k]

    # Step 3: 获取父文档（只取 top_k 条，减少 mget 开销）
    if return_full_content:
        knowledge_nos = [r["knowledge_no"] for r in search_results]
        parent_docs = get_parent_documents(knowledge_nos)
        for result in search_results:
            result["full_content"] = parent_docs.get(result["knowledge_no"], "")

    return search_results
```

### 3.6 Reranker 降级策略

`RERANKER_ENABLED = False` 时，系统自动退化为当前 RRF 行为，不影响任何现有功能。

## 三、方案三：动态阈值截断（Reranker 配套必做）

> **设计背景**：Reranker 和动态阈值是同一问题的上下半场。Reranker 负责给出有语义含义的 0~1 分数，动态阈值负责利用这个分数做智能截断。两者必须配套实现，否则 Reranker 的精排结果仍被硬编码 Top-K 浪费。

### 3.1 问题根因

```
排名  Reranker分数   当前处理
 1    0.97          ✅ 返回
 2    0.95          ✅ 返回
 3    0.91          ✅ 返回
 4    0.18          ❌ 不相关，但硬编码 Top-5 依然返回 → 送给 LLM 产生噪音
 5    0.12          ❌ 不相关，但硬编码 Top-5 依然返回 → 送给 LLM 产生噪音
```

硬编码 `top_k=5` 的问题：
- 只有 1 条相关时，引入 4 条噪音
- 有 8 条相关时，漏掉 3 条有用内容

### 3.2 算法：Elbow Method（肘部法则）

核心逻辑：**计算相邻分数的差值（斜率），差值超过阈值时判定为"断崖"，在断崖处截断**。

```
scores = [0.97, 0.95, 0.91, 0.18, 0.12]

相邻差值（斜率）：
  第1→2: 0.97 - 0.95 = 0.02  (正常波动)
  第2→3: 0.95 - 0.91 = 0.04  (正常波动)
  第3→4: 0.91 - 0.18 = 0.73  ← 超过阈值 0.3，断崖！截断在第3条之后
  第4→5: 0.18 - 0.12 = 0.06  (不再计算)

最终返回：前 3 条
```

### 3.3 新增方法设计

**在 `ESRetrievalService` 中新增私有方法**（无需新建文件）：

```python
def _dynamic_truncate(
    self,
    docs: List[Dict[str, Any]],      # Reranker 排序后的结果，必须含 rerank_score
    drop_threshold: float = 0.3,     # 触发截断的最小跌幅（可配置）
    min_results: int = 1,            # 至少返回几条（防止全截断）
    max_results: int = 5             # 最多返回几条（= top_k，防止过多）
) -> List[Dict[str, Any]]:
    """
    Elbow Method 动态截断：找到相邻 rerank_score 的断崖处截断。
    仅在 RERANKER_ENABLED=True 时调用（无 rerank_score 则无法截断）。
    """
    if len(docs) <= min_results:
        return docs

    scores = [d.get("rerank_score", 0.0) for d in docs]

    for i in range(len(scores) - 1):
        drop = scores[i] - scores[i + 1]
        if drop >= drop_threshold:
            cutoff = max(i + 1, min_results)
            logger.info(f"动态截断：在第{cutoff}条触发断崖（跌幅={drop:.3f}），从{len(docs)}条截断至{cutoff}条")
            return docs[:cutoff]

    # 未触发断崖，返回全部但不超过 max_results
    return docs[:max_results]
```

### 3.4 集成位置

紧接在 Reranker 精排之后，`get_parent_documents()` 之前：

```python
async def retrieve(
    self,
    query: str,
    top_k: int = 5,
    return_full_content: bool = True
) -> List[Dict[str, Any]]:

    # Step 1: 混合检索，多召回候选
    top_n: int = settings.RERANKER_TOP_N   # 50
    search_results: List[Dict[str, Any]] = await self.rrf_search(query, top_k=top_n)

    # Step 2: Reranker 精排 + 动态阈值截断（组合拳，不可拆分）
    if settings.RERANKER_ENABLED and search_results:
        search_results = await self.reranker_service.rerank(
            query, search_results, top_k=top_k
        )
        # ↑ Reranker 返回按 rerank_score 降序的 top_k 条

        if settings.DYNAMIC_THRESHOLD_ENABLED:
            search_results = self._dynamic_truncate(
                search_results,
                drop_threshold=settings.DYNAMIC_DROP_THRESHOLD,  # 默认 0.3
                min_results=settings.DYNAMIC_MIN_RESULTS,         # 默认 1
                max_results=top_k
            )
    else:
        search_results = search_results[:top_k]

    # Step 3: 获取父文档（只取动态截断后的实际数量）
    if return_full_content:
        knowledge_nos: List[str] = [r["knowledge_no"] for r in search_results]
        parent_docs: Dict[str, str] = self.get_parent_documents(knowledge_nos)
        for result in search_results:
            result["full_content"] = parent_docs.get(result["knowledge_no"], "")

    return search_results
```

### 3.5 新增配置项

```python
# config/settings.py 新增
DYNAMIC_THRESHOLD_ENABLED: bool = Field(default=False)   # 开关，默认关闭
DYNAMIC_DROP_THRESHOLD: float = Field(default=0.3)        # 断崖阈值，跌幅超过此值触发截断
DYNAMIC_MIN_RESULTS: int = Field(default=1)               # 最少返回条数，防止全截断
```

> **为什么默认关闭**：`drop_threshold=0.3` 是经验值，不同知识库的分数分布不同。建议上线后灰度观察分数分布，再决定是否开启和如何调参。

### 3.6 改动范围

```

  ├── _dynamic_truncate()    ← 新增私有方法
  └── retrieve()             ← 在 reranker 调用后接入 _dynamic_truncate

config/settings.py
  ├── DYNAMIC_THRESHOLD_ENABLED   ← 新增
  ├── DYNAMIC_DROP_THRESHOLD      ← 新增
  └── DYNAMIC_MIN_RESULTS         ← 新增
```

**不需要新增文件**，改动完全内聚在现有文件内。

### 3.7 降级策略

| 开关状态 | 行为 |
|---------|------|
| `RERANKER_ENABLED=false` | 动态截断不执行（无 rerank_score 可用），退化为 RRF + 硬编码截断 |
| `RERANKER_ENABLED=true` + `DYNAMIC_THRESHOLD_ENABLED=false` | Reranker 精排 + 硬编码 top_k 截断（V3.3 第一阶段行为） |
| `RERANKER_ENABLED=true` + `DYNAMIC_THRESHOLD_ENABLED=true` | 完整三级漏斗：召回 → 精排 → 动态截断 |

## 四、升级后完整检索流程

```
用户问题
    ↓
[Query Rewriting]            LLM 改写，生成原始 + 改写版本
    ↓
[multi_query_rrf_search]     双路并行检索
    ├── BM25 检索 × 2        (原始 + 改写)
    └── Native KNN 检索 × 2  (原始 + 改写，HNSW O(log n))  ← 方案一
    ↓
[RRF Fusion]                 倒数排名融合，k=60
    │                        输出：Top-50 候选 chunk
    ↓
[Cross-Encoder Reranker]     (query, chunk) 对语义精排      ← 方案二
    │                        服务商：硅基流动 bge-reranker-v2-m3
    │                        输出：Top-K，附 rerank_score（0~1）
    ↓
[Dynamic Thresholding]       Elbow Method 动态截断          ← 方案三（配套必做）
    │                        检测相邻分数断崖（跌幅 > 0.3）
    │                        输出：1~top_k 条（按实际相关性决定）
    ↓
[Get Parent Documents]       mget 批量取父文档完整内容（仅取实际截断后的条数）
    ↓
[LLM Generation]             流式输出最终答案
```

## 六、风险与注意事项

| 风险点 | 影响 | 应对措施 |
|--------|------|---------|
| ~~本地模型并发崩溃~~ | ~~CPU 100%，服务挂死~~ | **已规避：生产使用硅基流动 API，禁止主服务加载本地模型** |
| Reranker API 网络延迟 | +50~200ms | 控制候选集不超过 50 条；`RERANKER_ENABLED` 开关可随时关闭降级 |
| 硅基流动 API 调用费用 | 增加运营成本 | Rerank 调用量 = 查询量，按需评估；可设置 `RERANKER_TOP_N` 上限控制成本 |
| KNN `num_candidates` 设置过小 | 召回率下降 | 建议不低于 `k * 5`，初期设为 `k * 10` 观察效果 |
| 现有 `hybrid_search()` 未更新 | 该方法仍用 script_score | 主路径已是 `rrf_search()`，`hybrid_search()` 同步替换即可 |
| `DYNAMIC_DROP_THRESHOLD` 设置不当 | 过小：截断不足，噪音残留；过大：截断过度，有效内容丢失 | **默认关闭**，上线后灰度观察 rerank_score 分布，再确定合适阈值（建议先跑一周日志统计分位数） |
| 动态截断后返回 0 条 | LLM 无上下文，回答退化 | `DYNAMIC_MIN_RESULTS=1` 保底，至少返回 1 条得分最高的结果 |
